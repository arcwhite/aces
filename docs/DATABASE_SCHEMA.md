# Database Schema - Detailed Specification

## Overview

This document provides complete field-level specifications for all database tables in the Battletech: Aces Campaign Tracker.

## Timestamp Strategy

**All timestamps stored in UTC:**

The application follows Phoenix/Ecto best practices for timestamp handling:

### Database Layer
- **Type in migrations**: `:utc_datetime`
- **PostgreSQL column type**: `timestamptz` (timestamp with time zone)
- **Storage**: All timestamps stored as UTC in the database

### Application Layer
```elixir
# In Ecto schemas
field :inserted_at, :utc_datetime
field :updated_at, :utc_datetime

# Standard Phoenix timestamps macro
timestamps(type: :utc_datetime)  # Creates inserted_at and updated_at
```

### Display Layer
```elixir
# In LiveView/templates - convert to user's timezone
Calendar.strftime(sortie.inserted_at, "%Y-%m-%d %I:%M %p %Z",
  time_zone: user.timezone || "America/New_York")
```

**Naming Conventions:**
- `inserted_at`, `updated_at` - Standard Phoenix fields (auto-managed by Ecto)
- `*_at` suffix - Always UTC datetime (e.g., `completed_at`, `started_at`)
- No manual timezone conversion in business logic - keep everything UTC

## Fictional Timeline System

**BattleTech uses the Gregorian calendar**, so in-universe dates are simply far-future dates (3145, 3150, etc.).

### Two Parallel Timelines

Each campaign and sortie tracks **both real-world and fictional dates**:

| Purpose | Real-World | Fictional (In-Universe) |
|---------|------------|------------------------|
| **Company** | `inserted_at` (when created) | `founding_year` (e.g., 3145) |
| **Campaign** | `started_at`, `completed_at` | `fictional_start_date` (e.g., 3145-08-15) |
| **Sortie** | `date_played`, `completed_at` | `fictional_date` (e.g., 3145-09-03) |

### Database Implementation

```sql
-- Companies
founding_year INTEGER CHECK (founding_year >= 2500 AND founding_year <= 4000)

-- Campaigns
fictional_start_date DATE CHECK (fictional_start_date >= '2500-01-01')

-- Sorties
fictional_date DATE CHECK (fictional_date >= '2500-01-01')
```

### Validation Helper

```elixir
defmodule Aces.Campaigns do
  def calculate_days_between_sorties(sortie1, sortie2) do
    if sortie1.fictional_date && sortie2.fictional_date do
      Date.diff(sortie2.fictional_date, sortie1.fictional_date)
    end
  end

  def suggest_next_fictional_date(campaign, last_sortie) do
    base_date = last_sortie.fictional_date || campaign.fictional_start_date

    # Suggest 7-14 days after last mission
    Date.add(base_date, Enum.random(7..14))
  end
end
```

## Data Integrity Rules

### Company Units
- Cannot delete a company_unit that has deployments (restrict deletion)
- When unit is destroyed, create memorial entry automatically

### Pilots
- Cannot delete a pilot that has deployments (restrict deletion)
- When pilot is killed, create memorial entry automatically
- Pilot can only be assigned to one company_unit at a time

### Campaigns
- Warchest balance must always be >= 0
- Cannot delete campaign with sorties (restrict deletion)

### Sorties
- Cannot modify sortie after `is_completed = true` (application-level enforcement)
- Sum of deployed unit PVs cannot exceed `pv_limit` (application-level enforcement)

### Deployments
- Pilot and unit must belong to the same company
- Cannot deploy destroyed units
- Cannot deploy killed pilots
