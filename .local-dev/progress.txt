# Learnings
- When adding Phoenix LiveViews, agents have a tendency to implement ad-hoc methods to modify state in the DB. It's better to think of these as being an entire form, and committing to using a Changeset for validating user input.
- For complex migrations (like JSON to table), write a detailed spec first (see specs/PILOT_ALLOCATIONS.md). This captures context and decisions for future implementation.

# High-priority items
- [ ] On the Sortie Live view, when the Sortie is in-progress, we show the text "Crew: 100 SP" under any unit that has unnamed crew. Remove this. However, in the column "Pilot", under "Crew", show their default skill of 4.
- [ ] There is a viewer role for Companies. During testing, when I gave a user this role, they were still able to do a lot of actions. Please analyse our Company and Campaign views, and ensure that the viewer role can only read things. Where there are controls that they are not allowed to use, these controls should be visible but disabled.
- [ ] Controls on the campaign should be centralized, as well as relevant controls being available on the tabs they're concerned with. For example, the Overview tab should consolidate the 'complete campaign' button, hire pilot, purchase unit, and start/continue sortie buttons. "Purchase unit" should still be available on the Units tab.
- [ ] Wherever we display units, we should indicate their Size. We should do this as minimally as possible, but consistently ("SZ: 1" as an example).
